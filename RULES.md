# 📐 Правила взаимодействия при работе с кодом

## 0. Передача кода
- Всегда `cat > file << 'EOF'` (одинарные кавычки)
- Без `tee`, `echo "..." > file`, перенаправлений в одну строку

## 1. Разведка перед заменой
- `grep -n "ключ" файл` — найти точные строки
- Номера строк зафиксировать
- ❌ Не делать `sed` по паттернам без проверки

## 2. Самонаводящийся скрипт (основной метод)
- Один скрипт на всю операцию: поиск → проверка → правка → валидация
- Использовать `grep -q` для проверки маркера ПЕРЕД правкой
- Использовать `grep -q` для проверки дубликата ПЕРЕД вставкой
- `if/else` на каждый шаг — не продолжать при ошибке
- `python3 -m py_compile` после каждой правки
- При ошибке синтаксиса — `git checkout -- file` (откат)

## 3. Бэкап перед правками
- `cp файл data/backups/файл.bak.$(date +%Y%m%d_%H%M%S)`

## 4. Проверка после замены
- Python: `python3 -m py_compile файл.py`
- JSON: `python3 -m json.tool файл.json > /dev/null`
- Подсчёт функций: `grep -c "async def " файл.py`

## 5. Аудит
- Все функции/селекторы уникальны (count = 1)
- Скобки сходятся
- Синтаксис чистый

## 6. Откат
- `cp data/backups/файл.bak путь/к/файлу`
- Или `git checkout -- файл`

## 7. Git
- Коммит после каждой успешной правки
- `git push` после коммита
- ❌ Не пушить при ошибке синтаксиса

## 8. Data-driven подход (новое)
- Вопросы, меню, ответы — в JSON-конфигах
- Код — только движок, читающий конфиг
- Правки — через JSON, не через sed

## 9. Запрещено
- ❌ Создавать файлы без явной команды
- ❌ Дублировать функции/селекторы
- ❌ Переписывать файл целиком если можно заменить блок
- ❌ `sed` без проверки маркера
- ❌ Пушить битый код
